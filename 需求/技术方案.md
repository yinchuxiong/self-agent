# 个人工作助手 — 技术方案 v0.1

## 一、方案目标

本方案用于落地《个人工作助手 — 需求文档 v0.3》，建设一个长期运行在个人服务器上的多 Agent 协作系统。

核心目标：

- 基于 **LangGraph + LangChain** 实现 Supervisor + 多领域 Agent 的协作执行。
- 基于 **Skill / Tool / MCP / Workflow Registry** 实现能力统一注册、装配、启停、权限、统计。
- 基于 **FastAPI + React + Ant Design** 提供 Web UI、管理面板和实时对话。
- 基于 **PostgreSQL + Qdrant** 提供配置存储、执行日志、记忆系统和知识库检索。
- 基于 **APScheduler** 支持提醒、定时任务和定时 Workflow。
- 基于 **飞书 Bot** 支持外部消息入口。

默认技术决策：

| 项目 | 默认选择 |
|------|----------|
| 后端语言 | Python 3.11 |
| Agent 编排 | LangGraph |
| LLM 适配 | LangChain ChatModel，默认 DeepSeek API |
| 后端 API | FastAPI |
| 前端 | React + TypeScript + Ant Design |
| 实时输出 | SSE 优先，WebSocket 作为后续增强 |
| 关系存储 | PostgreSQL（生产）/ SQLite（本地开发） |
| 向量数据库 | Qdrant |
| 定时任务 | APScheduler |
| 部署 | Docker Compose |

### 1.1 当前确认与调整

| 事项 | 当前结论 |
|------|----------|
| MCP | 需要支持。第一期先完成 MCPRegistry 和接入框架，具体第三方 MCP 后续按配置接入 |
| 关系数据库 | 生产环境建议在服务器本地用 Docker Compose 部署 PostgreSQL；SQLite 仅作为开发或极简部署选项 |
| 向量数据库 | 默认推荐 Qdrant，服务器本地 Docker 部署；如后续数据规模极大再评估 Milvus |
| 工作目录 | 默认基于 Agent 部署目录/默认工作目录；Skill、Workflow 可在白名单内覆盖 |
| 高风险确认 | 采用“预览 + 确认令牌 + 分入口确认”的方式，Web UI 用弹窗，飞书用交互卡片，CLI 用命令行确认 |
| 调用明细保留 | 默认 90 天，超过后保留聚合统计，明细可归档 |
| Workflow 编排 | 第一期不做可视化 DAG 编排，使用 YAML + 表单编辑 |
| 日报细节 | 先预留日报数据模型、模板和 Workflow，具体内容规则后续补充 |
| 外部 Skill | 默认不可信，必须经过权限声明、依赖审查、沙箱运行、调用审计 |

## 四、Agent 设计

### 4.1 初始 Agent

| Agent | 定位 | 默认 Skill |
|-------|------|------------|
| Supervisor Agent | 请求级编排，不直接执行业务工具 | 无业务 Skill，仅有意图识别和编排 prompt |
| Programming Agent | 代码、Git、仓库、依赖、审查 | git-manager、code-reviewer、commit-helper、repo-doctor |
| Personal Tools Agent | 文件处理、格式转换、OCR、Excel/PDF/JSON | excel-master、pdf-master、json-master、format-shifter |
| Work Agent | 日报、飞书、会议、任务、日历 | daily-reporter、feishu-publisher、meeting-recorder、task-tracker |
| Scheduler Agent | 定时任务、提醒、周期巡检 | cron-master、reminder、health-watcher、schedule-auditor |

### 4.2 Agent 执行流程

```text
收到子任务
  → 加载 Agent 配置和默认上下文
  → 从 Memory Service 召回相关记忆
  → 扫描已装配 Skill
  → 匹配并激活 Skill
  → 构造 Agent prompt
  → 执行 LLM 推理
  → 调用 Tool / MCP
  → 写入调用日志
  → 返回结构化结果给 Supervisor
```

### 4.3 Agent 配置

```yaml
name: programming
display_name: Programming Agent
enabled: true
description: 负责代码、Git、仓库、依赖和代码审查任务
default_model: deepseek-chat
workspace_dir: E:/projects
allowed_paths:
  - E:/projects
permission_level: execute
equipped_skills:
  - git-manager
  - code-reviewer
  - commit-helper
  - repo-doctor
memory:
  enabled: true
  collection: programming
```

## 五、Capability Registry 设计

### 5.1 Registry 分层

| Registry | 管理对象 | 数据来源 |
|----------|----------|----------|
| AgentRegistry | Agent 元信息、启停、默认目录、权限上限 | 关系数据库 + 内置配置 |
| SkillRegistry | Skill 元信息、触发词、装配关系、版本 | YAML + 关系数据库 |
| ToolRegistry | 本地 Tool、参数 schema、权限、归属 Skill | Python 装饰器 + 关系数据库 |
| MCPRegistry | MCP Server、连接配置、暴露工具、状态 | 关系数据库 |
| WorkflowRegistry | Workflow DAG、触发方式、版本、执行策略 | YAML + 关系数据库 |

### 5.2 配置优先级

运行时配置按以下优先级合并：

```text
全局默认配置
  < Agent 配置
  < Skill 配置
  < Tool / MCP 配置
  < Workflow Step 配置
  < 单次请求显式参数
```

示例：如果 Agent 默认工作目录是 `E:/projects`，某个 Workflow 指定 `E:/projects/demo`，则该 Workflow 内步骤默认使用 `E:/projects/demo`。

### 5.3 热加载策略

- Skill YAML、Workflow YAML 支持热加载。
- Tool Python 代码变更第一期不做热加载，提示重启服务。
- MCP Server 配置修改后支持重新连接。
- 删除能力前必须做依赖检查。

## 六、Skill / Tool / MCP 运行机制

### 6.1 Skill

Skill 是业务能力包，包含：

- `skill.yaml`
- `prompt.md`
- `tools.py`
- `references/`

Skill 负责把 Tool/MCP 组织成对 Agent 有意义的能力。Agent 不应该直接面对一堆裸 MCP 工具。

### 6.2 Tool

Tool 是本地原子函数。

建议使用装饰器注册：

```python
@tool(
    name="git_diff",
    permission_level="read",
    timeout_seconds=30,
    owner_skill="git-manager",
)
def git_diff(repo_path: str, staged: bool = False) -> ToolResult:
    ...
```

Tool 执行前统一经过：

```text
参数校验
  → 权限检查
  → 工作目录检查
  → 高风险确认检查
  → 执行
  → 输出截断/脱敏
  → 写调用日志
```

### 6.3 MCP

MCP Server 由 MCPRegistry 管理。

配置示例：

```yaml
name: feishu-mcp
display_name: 飞书 MCP
enabled: true
transport: stdio
command: python
args:
  - -m
  - feishu_mcp_server
env_vars:
  FEISHU_APP_ID: ${FEISHU_APP_ID}
  FEISHU_APP_SECRET: ${FEISHU_APP_SECRET}
workspace_dir: E:/selfAgent
permission_level: external_publish
confirm_required: true
```

MCP 启动与工具刷新：

```text
保存 MCP 配置
  → 连接测试
  → 拉取 tools/list
  → 生成 MCP tool schema
  → 写入 ToolRegistry 的外部工具映射
  → 由 Skill 绑定使用
```

## 七、Workflow 技术设计

### 7.1 Workflow 定位

Workflow 是固定流程模板，适合沉淀可重复执行的多步骤任务。

示例：

```text
review 今日代码
  → 生成日报草稿
  → 用户确认
  → 发送飞书群
```

### 7.2 Workflow 执行模型

Workflow 使用 DAG 表达步骤依赖。第一期建议使用 YAML + 表单编辑，不优先做完整可视化 DAG 编辑器。

执行流程：

```text
加载 Workflow
  → 校验 DAG
  → 创建 trace_id
  → 按依赖执行 Step
  → 每个 Step 调用 Agent / Skill / Tool / MCP
  → 保存 Step 输出
  → 失败策略处理
  → 聚合最终结果
```

### 7.3 Step 类型

| 类型 | 说明 |
|------|------|
| agent | 调用某个 Agent，由 Agent 自己匹配 Skill |
| skill | 指定 Agent + Skill 执行 |
| tool | 直接调用本地 Tool，不推荐作为复杂业务主路径 |
| mcp_tool | 直接调用 MCP 暴露工具，高风险操作需确认 |
| condition | 条件判断 |
| confirm | 人工确认 |
| parallel | 并行分支 |

### 7.4 失败策略

| 策略 | 说明 |
|------|------|
| fail_fast | 任一步失败则终止 |
| retry | 按配置重试 |
| skip | 失败后跳过该步骤 |
| fallback | 调用备用步骤 |
| require_user_decision | 交给用户选择继续、重试、终止 |

## 八、请求处理链路

### 8.1 Web UI 对话请求

```text
POST /api/chat/sessions/{session_id}/messages
  → 创建 trace_id
  → 写入用户消息
  → Supervisor 意图识别
  → 判断是否匹配 Workflow
  → 简单任务：路由单 Agent
  → 复杂任务：拆解多个子任务
  → Agent 执行
  → 流式返回中间事件和最终答案
  → 写入记忆和统计
```

### 8.2 飞书 Bot 请求

```text
飞书事件回调
  → 校验签名
  → 解析用户和群聊上下文
  → 转为统一 ChatRequest
  → 进入 Supervisor
  → 流式结果转换为飞书消息
  → 必要时发送卡片或文件
```

### 8.3 Scheduler 请求

```text
APScheduler 触发
  → 加载 Job 配置
  → 调用 Workflow 或 Agent
  → 写执行日志
  → 失败时通知用户
```

## 九、记忆系统实现

### 9.1 短期记忆

存储在 LangGraph State 和关系数据库 session messages 中。

策略：

- 60% token 触发增量摘要。
- 80% token 触发滑动窗口。
- 95% token 触发紧急压缩。
- 工具输出超过阈值只保留摘要和引用。

### 9.2 中期记忆

会话结束后生成结构化摘要，写入关系数据库。

表结构建议：

```text
session_summaries
  id
  session_id
  summary
  key_decisions_json
  user_preferences_json
  unfinished_tasks_json
  created_at
```

### 9.3 长期记忆

结构化信息写关系数据库，语义信息写 Qdrant。

长期记忆类型：

- 用户画像
- 偏好记忆
- 实体记忆
- 经验记忆
- 任务记忆
- 知识片段

## 十、知识库实现

### 10.1 文档处理链路

```text
上传/导入文档
  → 识别文件类型
  → DocumentLoader 提取内容
  → Splitter 分片
  → 生成 embedding
  → 写入 Qdrant
  → 写入关系数据库文档元数据
```

生产环境中，文档元数据写入 PostgreSQL；本地开发环境可以使用 SQLite。

### 10.2 向量数据库选型

当前主流向量数据库可以按“个人长期服务是否好维护”来选择，而不是只看极限性能。

| 选型 | 特点 | 优点 | 局限 | 本项目建议 |
|------|------|------|------|------------|
| Qdrant | 专用向量数据库，Rust 实现，支持 payload 过滤、向量检索、混合检索能力 | Docker 部署简单，Python SDK 成熟，过滤能力强，适合个人服务器长期运行 | 需要单独维护一个服务 | **推荐默认使用** |
| Milvus | 面向大规模向量检索，分布式能力强 | 数据量很大时扩展性好，生态成熟 | 组件较多，个人服务器部署和运维偏重 | 后期数据量很大再考虑 |
| Weaviate | 向量数据库 + schema + 混合检索，功能完整 | 管理能力强，适合知识图谱/语义搜索类应用 | 部署和概念模型比 Qdrant 略重 | 可作为备选 |
| Chroma | Python 原生，开发体验轻 | 原型和本地开发很快，依赖少 | 长期服务、权限和运维能力偏轻 | 适合开发环境或轻量备选 |
| pgvector | PostgreSQL 扩展，直接在关系库里做向量检索 | 架构简单，一个 PostgreSQL 同时管业务数据和向量 | 专用向量能力、召回调优和大规模性能不如专用向量库 | 如果想极简部署可选 |
| LanceDB | 嵌入式/Serverless 风格，适合本地数据和多模态 | 文件型数据管理方便，轻量 | 生态和长期服务运维经验相对较新 | 暂不作为第一选择 |

**推荐结论**：

第一期采用 **PostgreSQL + Qdrant**：

- PostgreSQL：保存用户配置、Agent/Skill/Tool/MCP/Workflow 配置、会话、日志、统计、审计。
- Qdrant：保存知识库分片向量、长期记忆向量、语义召回数据。

这样比“只用 PostgreSQL + pgvector”多维护一个 Qdrant，但检索能力和后续扩展更稳；也比 Milvus 更轻，适合个人服务器。

### 10.3 Collection 设计

| Collection | 内容 |
|------------|------|
| common | 公共知识、用户偏好、通用规则 |
| programming | 编程相关文档、项目说明、代码规范 |
| work | 日报模板、会议纪要、工作文档 |
| tools | 文件处理模板、脚本说明 |
| scheduled | 定时任务配置、执行日志摘要 |

### 10.4 检索策略

默认使用混合检索：

```text
向量召回 top_k=10
  + BM25 关键词召回 top_k=10
  → 合并去重
  → metadata filter
  → rerank top_k=5
  → 注入 Agent 上下文
```

## 十一、数据库设计

生产环境默认使用 PostgreSQL，部署在同一台服务器上，通过 Docker Compose 管理。SQLite 仅用于本地开发、测试或极简单机模式。

选择 PostgreSQL 的原因：

- Web UI、飞书 Bot、Scheduler、后台任务会同时读写，PostgreSQL 比 SQLite 更适合长期并发运行。
- 调用日志、统计聚合、审计记录会持续增长，PostgreSQL 更适合后续查询和归档。
- 后续如需要，也可以增加 pgvector 扩展作为备用向量能力。
- 数据库仍然是“本地数据库”，不需要使用云数据库。

## 十二、API 设计

### 12.1 对话 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/chat/sessions` | 创建会话 |
| GET | `/api/chat/sessions` | 会话列表 |
| GET | `/api/chat/sessions/{id}` | 会话详情 |

### 12.2 Agent / Skill / Tool / MCP / Workflow API

| 方法 | 路径 | 说明 |

## 十四、安全方案

### 14.1 权限等级

| 等级 | 能力 |
|------|------|
| read | 读取文件、查询状态、检索知识库 |
| write | 写入文件、更新本地配置 |
| execute | 执行 shell、Git 操作、本地脚本 |
| external_publish | 发送飞书消息、发布文档、发送邮件 |
| dangerous | 删除文件、推送 Git、写数据库、批量修改外部系统 |

### 14.2 确认策略

默认需要确认：

- 删除文件
- 执行 shell
- Git push
- Git branch delete
- 写入数据库
- 飞书发送到群聊
- 发布或更新远程文档
- 定时任务启停

确认机制采用“先预览，再授权执行”：

```text
Agent/Workflow 生成执行计划
  → PermissionService 判断是否高风险
  → 生成 ConfirmationRequest
  → 按入口发送确认请求
  → 用户确认后生成一次性 confirmation_token
  → Tool/MCP/Workflow Step 带 token 执行
  → 写入 confirmations 和 audit_logs
```

#### 不同入口的确认方式

| 入口 | 确认方式 |
|------|----------|
| Web UI | 弹窗或抽屉展示操作摘要、影响范围、参数、diff/预览，用户点击确认 |
| 飞书 Bot | 发送交互卡片，包含操作摘要、目标对象、确认/取消按钮 |
| CLI | 展示操作摘要，要求输入 `yes` 或一次性确认码 |
| Scheduler | 默认不能静默执行高风险操作；必须提前配置“预授权规则”或转为待确认任务 |

#### ConfirmationRequest 字段

```text
id
trace_id
entrypoint
risk_level
operation_type
target_summary
preview_text
requested_by
expires_at
status: pending | approved | rejected | expired
approved_at
confirmation_token_hash
```

#### 预授权规则

为了避免每天重复确认低风险固定任务，可以支持预授权，但只允许细粒度配置：

- 指定 Workflow。
- 指定工作目录。
- 指定目标对象，例如某个飞书个人会话或测试群。
- 指定有效期。
- 指定最大调用次数或每天最多次数。

高风险 destructive 操作不允许长期预授权，例如删除文件、写数据库、Git push 到主分支。

### 14.3 脱敏策略

日志与统计中必须脱敏：

- API Key
- Token
- Secret
- Password
- Cookie
- 飞书 App Secret
- 数据库连接密码

### 14.4 外部 Skill 沙箱机制

外部 Skill 默认视为不可信。系统需要区分“纯配置/Prompt 型 Skill”和“可执行代码型 Skill”。

| Skill 类型 | 风险 | 沙箱要求 |
|------------|------|----------|
| Prompt-only Skill | 只包含 prompt、triggers、references | 可直接装配，但 references 读取仍受路径白名单限制 |
| Tool Skill | 包含 Python Tool 代码 | 必须经过依赖审查和沙箱执行 |
| MCP Skill | 绑定外部 MCP Server | MCP Server 也必须经过权限配置和连接审计 |
| Workflow Skill | 包含流程模板 | 每个 Step 仍按 Agent/Skill/Tool/MCP 权限执行 |

#### 外部 Skill 安装流程

```text
导入 Skill 包
  → 读取 manifest
  → 展示权限声明
  → 检查依赖和入口文件
  → 静态扫描风险项
  → 用户确认安装
  → 放入隔离目录
  → 默认禁用
  → 管理员手动启用并分配权限
```

#### Skill manifest 必填权限声明

```yaml
name: external-example
version: 0.1.0
type: tool-skill
permissions:
  filesystem:
    read:
      - "${WORKSPACE_DIR}"
    write: []
  network:
    enabled: false
    allow_hosts: []
  shell:
    enabled: false
  env:
    allow:
      - "SAFE_ENV_NAME"
  mcp:
    allow_servers: []
runtime:
  sandbox: container
  timeout_seconds: 60
  memory_limit_mb: 512
  cpu_limit: "0.5"
```

#### 沙箱执行策略

生产环境建议把外部代码型 Skill 放到独立容器或受限进程中执行：

- 默认无网络；需要网络时必须声明 allowlist host。
- 默认只读挂载 Skill 目录。
- 工作目录按需挂载，且只能挂载白名单路径。
- 环境变量按 allowlist 注入，敏感变量默认不传。
- 限制 CPU、内存、执行时长、输出大小。
- 禁止访问宿主 Docker socket。
- 禁止默认继承宿主用户权限。
- 工具输出必须经过脱敏和截断。

第一期可采用两级实现：

| 阶段 | 实现 |
|------|------|
| MVP | 外部 Skill 默认禁用；启用前做权限声明、路径白名单、依赖检查；执行时用受限 subprocess |
| 加固版 | 外部 Tool Skill 统一在 Docker 沙箱容器中执行，按调用挂载临时目录和工作目录 |

#### 风险扫描项

导入外部 Skill 时至少扫描：

- 是否调用 `subprocess`、`os.system`、`shutil.rmtree`、`socket`、`requests` 等高风险 API。
- 是否读取环境变量。
- 是否包含安装脚本。
- 是否声明网络访问。
- 是否要求写入工作目录外路径。
- 是否绑定高权限 MCP。

扫描不能替代沙箱，只用于给用户展示风险。

#### 外部 Skill 审计

外部 Skill 的所有行为必须记录：

- 安装来源。
- manifest 权限声明。
- 启用/禁用记录。
- 每次 Tool/MCP 调用。
- 文件读写摘要。
- 网络目标摘要。
- 失败和异常信息。

## 十八、关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| Agent 过度拆分 | 系统复杂度上升 | 初始只保留 4 个领域 Agent，优先扩 Skill |
| MCP 裸露过多工具 | 调用不可控，权限难管 | MCP 必须由 Skill 包装，高风险默认确认 |
| 工具输出过大 | 撑爆上下文 | 输出截断、摘要化、引用文件 |
| 数据库日志增长 | 统计和日志增长快 | 明细保留 90 天 + 聚合统计长期保留 + 定期归档 |
| Workflow 配置复杂 | 用户难编辑 | 第一期 YAML + 表单，后续再做 DAG 可视化 |
| 飞书误发 | 外部影响大 | 群发和文档发布默认预览确认 |
| Shell/Git 高风险 | 破坏本地环境 | 工作目录白名单 + 权限等级 + 确认 |
